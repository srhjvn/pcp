# Service Framework

## Purpose

This file defines the service-layer structure of the Personal Context Portfolio (PCP).  
It translates the portfolio from a knowledge repository into a usable service system by clarifying:
- what services are offered
- who they are for
- what problems they solve
- what outputs they produce
- which source files support delivery
- what is ready now vs what still needs to be built

This file is intended to be living documentation.

---

## Phase 2 Objective: Service Enablement

Phase 2 establishes the service foundation needed to make the portfolio operationally useful in client, project, or scoped support contexts.

The goal is to define a lean, repeatable service structure that can support:

- clear offer articulation
- target audience identification
- consistent problem/solution language
- output-to-source-file mapping
- asset gap identification
- intake and routing workflows

---

## Service Design Principles

The service structure should follow these rules:

- Keep offers simple and understandable.
- Favor modular service components over overly custom descriptions.
- Reuse existing source files wherever possible.
- Apply templates where available for consistency.
- Treat all service documentation as living documentation.
- Separate "ready now" offers from "partially enabled" offers.
- Make it easy to identify what is missing before a service is presented externally.

---

## Service Categories

The initial service structure is organized into three primary categories.

### 1. Strategic Clarity

Services in this category help a person, founder, consultant, or small business move from scattered thinking, fragmented material, or unclear positioning into clearer structure and usable direction.

Typical focus areas include:

- positioning refinement
- offer clarity
- audience/problem definition
- operating system clarity
- decision support synthesis

### 2. Documentation and Translation Support

Services in this category help convert raw inputs, notes, experience, or operational knowledge into structured documentation and usable language.

Typical focus areas include:

- structured write-ups
- workflow drafting
- process documentation
- requirements shaping
- translation of complex material into practical documents

### 3. Personal Context Portfolio Enablement

Services in this category help build, refine, maintain, or operationalize the PCP system itself or a similar modular documentation system.

Typical focus areas include:

- PCP structure refinement
- service layer development
- README and navigation clarity
- workflow and governance setup
- maintenance guidance

---

## Service Offer Structure

Each service offer should be defined using the following structure:

- **Service Name**
- **Category**
- **Primary Audience**
- **Situational Trigger**
- **Core Problem**
- **Service Response**
- **Typical Outputs**
- **Primary Source Files**
- **Template Support**
- **Current Readiness**
- **Notes / Gaps**

## Target Audience Framework

Target audiences should be defined at the offer level using the following fields:

- **Primary Audience** = who this service is mainly for
- **Situational Trigger** = what is happening that causes them to seek help
- **Buying Motivation** = why this matters now
- **Fit Signals** = indicators the service is a strong match
- **Non-Fit Signals** = indicators the request should be routed elsewhere

### Core Audience Groups

#### 1. Solo Consultants and Independent Professionals
Need clearer positioning, reusable language, structured offers, or support organizing complex experience into usable materials.

#### 2. Small Business Operators and Founders
Need lightweight systems, process clarity, cleaner internal documentation, or a more coherent operating structure.

#### 3. Professionals in Transition
Need help translating experience, clarifying value, organizing goals, or creating materials that support a next-step shift.

#### 4. Modular-System Builders
Need a PCP-like structure to organize personal, strategic, or service-related context in a maintainable way.

## Problem / Solution Language Standard

All service offers should describe the work using the same basic structure.

### Problem
What is difficult, unclear, scattered, blocked, or under-structured right now?

### Current State
What is happening now because the problem has not been solved?

### Service Response
What does the service actually do in practical terms?

### Output
What tangible document, structure, or system does the client receive?

### Result
What becomes easier, clearer, faster, or more repeatable afterward?

---

## Standard Problem / Solution Language

### Base Problem Statement
The information, experience, or work is real, but it is too scattered, unclear, or under-structured to be consistently useful.

### Base Current State
Important decisions, communication, or service delivery are slowed down because the material is not yet organized into a usable system.

### Base Service Response
The service organizes raw context into clearer structure, better language, and practical outputs that can be reused.

### Base Output Statement
The client receives documents, workflow structures, language blocks, or service framing they can actually use.

### Base Result Statement
The work becomes easier to explain, easier to maintain, and easier to move forward without rebuilding from scratch each time.

## Readiness Status Definitions

Use the following readiness markers across all service offers.
### Ready
The service is clearly defined, supported by source files, and can be delivered now with available materials.
### Ready to Pilot
The service can be used in a limited or flexible way, but may still need stronger templates or examples.
### Partially Ready
The service direction is clear, but important assets, templates, or delivery supports are still missing.
### Not Ready
The service concept exists, but it should not yet be presented externally as an active offer.
