# Personal Context Portfolio

**Status:** Active  
**Last Updated:** 2026-04-06  
**Purpose:** Explain the purpose, structure, and maintenance approach for the Personal Context Portfolio repository.

## Purpose
The Personal Context Portfolio (PCP) is a modular set of markdown files designed to capture durable personal and professional context in a portable, reusable format.

This repository exists to reduce the need to repeatedly reconstruct important context from scratch. It is intended to support clearer writing, better decisions, stronger continuity across projects and platforms, and more efficient drafting over time.

The PCP is meant to function as living documentation, not static information.

## Design Principles
This repository is built around a few core principles:

- Markdown first for portability and platform flexibility
- Modular structure so context can be updated one file at a time
- Consistency through templates where useful
- Living documentation that evolves as work, priorities, and context change
- Practical reuse across planning, writing, positioning, decision-making, and collaboration

## Repository Goals
The goals of this repository are to:

- document important personal and professional context in a structured way
- create reusable reference material for future writing and planning
- improve continuity across projects, tools, and opportunities
- make context easier to carry across platforms
- reduce friction in communication, drafting, and decision-making
- support work that is more coherent, portable, and self-defined over time

## File Purpose Summary

### `Identity.md`
Captures durable identity-level context, including professional identity, strengths, values, current season, and future direction.

### `RolesandResponsibilities.md`
Documents recurring role patterns, actual operating responsibilities, and the distinction between formal title and real contribution.

### `GoalsAndPriorities.md`
Defines what matters most right now, what is being optimized for, and how progress is understood in the current season.

### `CurrentProjects.md`
Tracks active workstreams, current focus areas, and how ongoing projects relate to one another.

### `TeamAndRelationships.md`
Documents typical working relationships, collaboration patterns, stakeholder dynamics, and relational friction points.

### `ToolsAndSystems.md`
Captures the tools, platforms, system environments, and working artifacts that shape day-to-day operating reality.

### `CommunicationStyle.md`
Defines communication preferences, drafting style, collaboration tone, and what makes communication more or less effective.

### `PreferencesAndConstraints.md`
Makes practical operating preferences explicit, including always-do guidance, avoid guidance, friction points, and active constraints.

### `DomainKnowledge.md`
Captures applied subject-matter knowledge, especially where operations, systems, workflows, and business reality intersect.

### `DecisionLog.md`
Provides a structured record of meaningful decisions, including rationale, tradeoffs, constraints, and revisit triggers.

## Naming Standard
To keep the repository consistent:

- use clear, descriptive file names
- use `.md` for all core files
- use `TitleCase` for primary portfolio documents
- avoid spaces in file names
- keep file names stable once in active use unless there is a strong reason to rename them

Current naming pattern:
- `Identity.md`
- `RolesandResponsibilities.md`
- `GoalsAndPriorities.md`
- `CurrentProjects.md`

Examples for future supporting files:
- `ServiceOfferings.md`
- `WritingThemes.md`
- `OperatingPrinciples.md`
- `OpportunityPipeline.md`
