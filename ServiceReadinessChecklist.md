# Service Readiness Checklist

## Purpose

This file provides a standard way to evaluate whether a service offer is ready to use, ready to pilot, only partially ready, or not ready.

It is intended to support more consistent service decisions by making it easier to check:
- whether the offer is clearly defined
- whether the audience is clear
- whether the problem/solution language is usable
- whether outputs are known
- whether source files are mapped
- whether templates exist
- whether missing assets are understood
- whether intake and routing can support the service

This file is living documentation and should be updated as the service layer matures.

---

## How to Use This File

Use this checklist when:

- a new service offer is introduced
- an existing service offer is being reviewed
- a service is being prepared for pilot use
- a service is being considered for active external presentation
- the team needs a quick readiness snapshot

This checklist should be used alongside:

- `ServiceFramework.md`
- `ServiceAssetMap.md`
- `ServiceIntakeWorkflow.md`

---

## Readiness Status Options

Use one of the following status levels after completing the checklist.

### Ready
The service is clearly defined, appropriately supported, and can be delivered now with acceptable consistency.

### Ready to Pilot
The service can be delivered in a limited or flexible form, but still depends on judgment, adaptation, or incomplete support assets.

### Partially Ready
The service direction is clear, but one or more important assets, templates, or workflow supports are still missing.

### Not Ready
The service should not yet be treated as operational.

---

## Readiness Evaluation Method

Review the service against the checklist below.

Mark each item as one of the following:

- **Yes** = clearly in place
- **Partial** = somewhat in place, but still incomplete
- **No** = not in place

After completing the review, use the interpretation guide to determine the readiness status.

---

## Core Readiness Checklist

### 1. Offer Definition
- [ ] The service has a clear name.
- [ ] The service category is defined.
- [ ] The service scope is understandable.
- [ ] The service is meaningfully distinct from other offers.
- [ ] the primary use case is clear.

### 2. Audience Definition
- [ ] The primary audience is identified.
- [ ] The situational trigger is defined.
- [ ] The buying motivation is understandable.
- [ ] fit signals are visible.
- [ ] non-fit signals are visible.

### 3. Problem / Solution Language
- [ ] The core problem is clearly stated.
- [ ] The current-state description is usable.
- [ ] The service response is understandable.
- [ ] The output statement is concrete.
- [ ] the result after delivery is clear.

### 4. Outputs and Deliverables
- [ ] The typical outputs are identified.
- [ ] The outputs are realistic for the service.
- [ ] The outputs are consistent with the stated problem.
- [ ] The outputs are specific enough to support scoping.
- [ ] there is no major confusion about what the client receives.

### 5. Source File Support
- [ ] Primary source files are mapped.
- [ ] Supporting source files are mapped.
- [ ] The mapped files are relevant to the service.
- [ ] The source material is sufficient to support delivery.
- [ ] dependencies are known.

### 6. Template and Asset Support
- [ ] Required templates have been identified.
- [ ] Existing templates are usable.
- [ ] Missing templates are known.
- [ ] Missing assets are tracked.
- [ ] the service is not overly dependent on ad hoc drafting.

### 7. Intake and Workflow Support
- [ ] The service can be classified during intake.
- [ ] The service can be fit-checked during intake.
- [ ] The service can be routed using current workflow logic.
- [ ] support status can be assessed.
- [ ] next-step scope can be defined without major confusion.

### 8. Delivery Readiness
- [ ] The service could be explained simply to an internal reviewer.
- [ ] The service could be initiated with currently available materials.
- [ ] The service could be delivered with reasonable consistency.
- [ ] likely risks or gaps are visible.
- [ ] the service is appropriate for current priorities.

---

## Readiness Interpretation Guide

Use the checklist results to assign a working status.

### Ready
Use **Ready** when:
- nearly all items are marked Yes
- any Partial items are minor
- no major structural blockers remain
- the service can be delivered now with acceptable repeatability

### Ready to Pilot
Use **Ready to Pilot** when:
- the core service structure is in place
- the service can be delivered now
- but some asset, example, or workflow gaps still exist
- and delivery still depends partly on judgment rather than full enablement

### Partially Ready
Use **Partially Ready** when:
- the service concept is valid
- the audience and problem are generally clear
- but multiple support elements are still missing
- or the service would still rely heavily on one-off setup

### Not Ready
Use **Not Ready** when:
- the service is not yet clearly defined
- the audience or problem is still unclear
- outputs are not stable
- source-file support is weak
- or intake and delivery logic are not usable yet

---

## Lean Decision Rule

If a service cannot yet answer these questions clearly, it is not Ready:

- What is the service?
- Who is it for?
- What problem does it solve?
- What does the client receive?
- What source material supports delivery?
- What template or workflow support exists?
- What happens when a request comes in?

## Priority Gating Questions

Before presenting a service externally, confirm the answer to each of these is yes:

- [ ] The service can be described in plain language.
- [ ] The audience can recognize themselves in the description.
- [ ] The problem and output are both concrete.
- [ ] the service is supported by actual source material.
- [ ] the current workflow can intake and route the request.
- [ ] missing assets will not undermine delivery quality.
- [ ] the service fits current priorities and operating direction.

If several of these remain unchecked, the service should stay internal or pilot-only.

## Definition of Done

This checklist is working correctly when:

- service readiness can be evaluated consistently
- different offers can be compared using the same criteria
- major blockers become visible quickly
- the difference between Ready, Pilot, Partial, and Not Ready is usable in practice
- the checklist supports real service activation decisions rather than just documentation
